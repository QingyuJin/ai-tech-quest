# Product Strategy

## Product Name

Product Strategy Layer for AI Tech Quest.

## Problem

Qingyu's work spans AI applications, RAG systems, Unity interaction, ML experiments, and web product development. Without a product strategy layer, these projects can look like separate demos instead of a coherent portfolio ecosystem.

## Target Users

- Qingyu, as the portfolio owner and product builder.
- Interviewers evaluating technical direction.
- Clients evaluating whether the work can become paid products.
- Future collaborators who need to understand the product map quickly.

## Core Features

- Product positioning for each portfolio product.
- Repo and naming strategy.
- Case study planning.
- Demo and deployment planning.
- Monetization notes for freelance-ready products.
- Priority map between showcase, revenue, and research projects.

## Tech Stack

- Markdown documentation.
- GitHub repository structure.
- Product roadmaps.
- Portfolio copywriting.
- Case study templates.

## MVP Scope

- Keep AI Tech Quest as the main product.
- Define AI Business Assistant as the first monetizable service product.
- Define TW Civic RAG as the high-technical-depth AI product.
- Define BuildFlow as the productized direction connected to `qingyu-web-studio`.
- Keep ML Experiment Lab as technical research support.
- Keep older practice repositories in background skills only.

## Monetization

This layer supports monetization by clarifying which projects can become client offers:

- AI Business Assistant for small business automation.
- BuildFlow for landing pages and small web product delivery.
- RAG demos for document QA consulting and technical interviews.

## Roadmap

1. Finalize product positioning and folder structure.
2. Write one-page case study templates for each product.
3. Replace demo placeholders with deployed URLs.
4. Create pricing and delivery packages for AI Business Assistant and BuildFlow.
5. Convert strongest products into standalone public repos when ready.
